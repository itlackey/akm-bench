# AKM Benchmark Operator's Guide

This document is the detailed reference for running and operating `akm-bench`.
The quickstart lives in `README.md`.

## Repo Layout

| Path | Purpose |
| --- | --- |
| `src/` | Benchmark harness and CLI |
| `config/` | Run configs and opencode config fixtures |
| `fixtures/corpus/` | Benchmark tasks and workflow specs |
| `fixtures/stashes/` | Reusable stash fixtures |
| `results/` | Saved benchmark reports and baselines |
| `tests/` | Harness tests |
| `docs/` | Operator documentation |

Task fixtures live under `fixtures/corpus/tasks/<domain>/<task-id>/`.
Workflow specs live under `fixtures/corpus/workflows/`.
Committed opencode config fixtures live under `config/opencode.json`.

## Running Benchmarks

Install dependencies:

```sh
bun install
```

Run the quick benchmark:

```sh
bun run src/cli.ts config/nano-quick.json
```

Run the full benchmark:

```sh
bun run src/cli.ts config/full.json
```

Override seeds or parallelism:

```sh
bun run src/cli.ts config/nano-quick.json --seeds 3 --parallel 2
```

Checked-in run configs:

- `config/nano-quick.json`
- `config/full.json`
- `config/failing-tasks.json`
- `config/curate-test.json`

The run-config schema lives at `config/bench-run-config.schema.json`.

## Opencode Config Discovery

Opencode config resolution order is:

1. `--opencode-config <path>`
2. `BENCH_OPENCODE_CONFIG`
3. `config/opencode.local.json`
4. `config/opencode.json`

Model resolution order is:

1. `BENCH_OPENCODE_MODEL`
2. run config `model`
3. opencode config `model`

Use a custom opencode config for one run:

```sh
bun run src/cli.ts config/nano-quick.json --opencode-config /path/to/opencode.json
```

Use environment overrides:

```sh
BENCH_OPENCODE_CONFIG=config/opencode.local.json \
BENCH_OPENCODE_MODEL=local/qwen/qwen3.5-9b \
bun run src/cli.ts config/nano-quick.json
```

The harness does not auto-discover a global opencode config. Use
`BENCH_OPENCODE_CONFIG` or `--opencode-config` if you want to opt into a
non-repo config file.

The bench reads only the source config's `model` and `provider` fields. The
per-run `opencode.json` is still generated by the harness so bench-controlled
`permission` and `plugin: []` invariants stay in force.

## Local Providers

### LM Studio

1. Install and open LM Studio.
2. Download a model you want to benchmark.
3. Start the local server.
4. Confirm the OpenAI-compatible endpoint is available at `http://127.0.0.1:1234/v1`.
5. Write your local config to `config/opencode.local.json`.

Example:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "model": "lmstudio/qwen/qwen3.5-9b",
  "provider": {
    "lmstudio": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "LM Studio",
      "options": {
        "baseURL": "http://127.0.0.1:1234/v1",
        "timeout": 600000
      },
      "models": {
        "qwen/qwen3.5-9b": {
          "name": "Qwen3.5 9B",
          "limit": {
            "context": 32768,
            "output": 8192
          },
          "capabilities": {
            "tool": true
          }
        }
      }
    }
  }
}
```

### Ollama

1. Install Ollama.
2. Start the Ollama server.
3. Pull the model you want to benchmark.

```sh
ollama pull qwen3.5:9b
```

4. Use the OpenAI-compatible endpoint at `http://127.0.0.1:11434/v1`.
5. Write your local config to `config/opencode.local.json`.

Example:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "model": "ollama/qwen3.5:9b",
  "provider": {
    "ollama": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Ollama",
      "options": {
        "baseURL": "http://127.0.0.1:11434/v1",
        "timeout": 600000
      },
      "models": {
        "qwen3.5:9b": {
          "name": "Qwen3.5 9B",
          "limit": {
            "context": 32768,
            "output": 8192
          },
          "capabilities": {
            "tool": true
          }
        }
      }
    }
  }
}
```

Model strings must match `<provider-key>/<model-key>`.

Examples:

- `lmstudio/qwen/qwen3.5-9b`
- `ollama/qwen3.5:9b`

Quick troubleshooting:

- if the run fails immediately, confirm the local server is listening on the expected port
- if the model responds but performs poorly, try a stronger tool-capable model
- if you keep multiple configs around, prefer `--opencode-config` over editing the committed fixture

## Reports

Persistent run artifacts are written under `results/` by default.

Compare two saved reports:

```sh
bun run src/cli.ts compare --base results/baseline.json --current /tmp/current-report.json
```

Compute per-asset attribution:

```sh
bun run src/cli.ts attribute --base /tmp/utility-report.json --top 5
```

## Tmp Root

All harness tmp directories live under `${AKM_CACHE_DIR}/bench/`, not `/tmp`.
Use `benchTmpRoot()` and `benchMkdtemp()` from `src/tmp.ts` for harness tmp
directories.

## Test Scope

The default verification loop is:

```sh
bun run lint
bun run typecheck
bun test ./tests
```

Spawned CLI integration tests in `tests/cli.test.ts` are skipped by default.
Opt in with:

```sh
AKM_BENCH_RUN_CLI_TESTS=1 bun test ./tests/cli.test.ts
```
